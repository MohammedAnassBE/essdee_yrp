## Root cause of the IPD create-500

The /web IPD create 500 is frappe.exceptions.QueryDeadlockError (MariaDB 1213) raised on the parent INSERT INTO `tabItem Production Detail` (Error Log vlga15qfle, POST /api/resource/Item Production Detail, doc TT-CLOTH-14, fully rolled back). It is layout-independent (Demo 7 is irrelevant; the SPA sends byte-identical payloads under both layouts and the failure is below any layout code). Two defects combine: (1) ItemProductionDetail.autoname (apps/yrp/yrp/yrp/doctype/item_production_detail/item_production_detail.py) computed the PK as f"{item}-{max(version)+1}" from a plain non-locking SELECT with no serialization, so the create's name choice is race-prone — any concurrent transaction (a second in-flight save of the same item computing the identical PK, or, in the 10:30 incident, the long-lived uncommitted `bench run-tests` transaction running against the live site) turns the parent INSERT into a lock wait; because essdee_yrp.ipd_validations.create_new_mapping_values has already written ~7 Item Item Attribute Mapping rows inside the same transaction before that INSERT (traceback: transaction_writes=7), the create holds locks its counterparty can want, InnoDB detects a cycle and kills the create (it has the fewest undo entries). (2) Nothing in the stack performs MariaDB's own prescribed remedy for 1213 ("try restarting transaction" — safe, since the rollback is total): frappe core does not retry web requests and QueryDeadlockError is a bare Exception with no http_status_code and no _server_messages, so client.js surfaced it as the opaque "Request failed with status 500" toast. Fix constraints verified on this server: isolation is REPEATABLE-READ (so the waiter's max(version) re-read MUST be a locking current-read or it reuses its stale snapshot and still picks the duplicate name), autoname runs before the insert transaction's first write (so a waiter at the Item-row lock holds nothing — no cycle possible), and `item` had no index (so the locking read needed one to avoid X-locking the whole table).

## Fix applied (uncommitted)

CHANGES. (1) item_production_detail.py autoname: before computing the version it now X-locks the parent Item row (frappe.db.get_value("Item", self.item, "name", for_update=True)) — autoname is the transaction's first lock/write point, so a concurrent same-item create parks there holding no locks (deadlock cycle impossible) until the first commits; the max(version) SELECT now ends in FOR UPDATE — a locking current-read, required under this server's REPEATABLE-READ isolation because after the lock wait a plain SELECT would reuse the transaction's stale snapshot, miss the just-committed version, and still produce the duplicate PK. Insert() order verified in frappe v16: _validate_links then set_new_name run before run_before_save_methods, so the Item exists at lock time and no writes precede the wait. (2) item_production_detail.json: "search_index": 1 on the item field (+ modified bumped to 2026-07-16 12:00:00) so the locking read scans only that item's index range instead of X-locking all ~297 rows; the index was applied to the live DB immediately via frappe.db.add_index("Item Production Detail", ["item"]) (verified via SHOW INDEX: item_index) — bench migrate was NOT run (forbidden); a future migrate is a no-op for this index (add_index is idempotent).
